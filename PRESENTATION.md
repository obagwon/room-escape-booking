# 고급자바 프로그래밍 기말 프로젝트 발표 자료

## 1. 5분 발표 버전

### 1) 프로젝트 제목
**방탈출 예약 관리 시스템**입니다.  
기존 Room Booking 구조를 기반으로, 방탈출 카페 운영에 필요한 고객, 지점, 테마, 예약, 플레이 결과, 통계 기능을 Java Swing과 CLI로 구현했습니다.

### 2) 개발 배경과 문제 정의
방탈출 카페는 예약 시간, 테마, 인원, 가격, 플레이 결과, 힌트 사용량을 함께 관리해야 합니다.  
단순히 “방이 예약되었는지”만 저장하면 다음과 같은 문제가 생깁니다.

- 같은 날짜와 같은 테마에 시간이 겹치는 예약을 막기 어렵습니다.
- 테마별 권장 인원과 가격을 반영하기 어렵습니다.
- 플레이가 끝난 뒤 성공 여부와 힌트 사용 횟수를 기록하기 어렵습니다.
- 운영자가 전체 예약 수, 취소 수, 매출, 인기 테마를 확인하기 어렵습니다.

### 3) 해결 목표
이 프로젝트의 목표는 방탈출 카페 운영자가 사용할 수 있는 데스크톱 예약 관리 프로그램을 만드는 것입니다.

- 고객과 관리자 역할 구분
- 방탈출 지점과 테마 관리
- 예약 생성, 취소, 조회
- 예약 중복 방지
- 플레이 결과와 힌트 사용량 기록
- 파일 기반 저장/불러오기
- Stream API 기반 운영 통계 제공
- GUI와 CLI의 한글 사용자 인터페이스 제공

### 4) 사용 기술
이 프로젝트에서는 고급자바 프로그래밍 수업에서 배운 개념을 다음과 같이 적용했습니다.

| 수업 개념 | 적용 위치 |
| --- | --- |
| Java 17 | 전체 프로젝트 |
| Swing GUI | `src/view/GUI` |
| CLI | `src/view/CLI/CommandLine.java` |
| MVC 구조 | `model`, `service`, `controller`, `view` 패키지 분리 |
| Collection Framework | `Map`, `HashMap` 기반 데이터 저장 |
| Enum | `UserRole`, `ReservationStatus` |
| File I/O | 각 Service의 `save/load` 메서드 |
| Object Serialization | `ObjectOutputStream`, `ObjectInputStream` |
| Stream API | `StatisticsService` 통계 계산 |
| 예외 처리 | 입력 검증 실패, 저장/불러오기 실패 처리 |
| JUnit | `src/Tests/basic` 테스트 코드 |

### 5) 전체 구조
프로젝트는 MVC 구조를 따릅니다.

- **Model**: 데이터 표현  
  `User`, `Room`, `Reservation`, `PlayResult`, `Building`
- **Service**: 비즈니스 로직과 저장/조회/검증 담당  
  `UserService`, `RoomService`, `ReservationService`, `PlayResultService`, `StatisticsService`
- **Controller**: View와 Service 사이 요청 전달  
  `BookingResource`
- **View**: 사용자 화면  
  Swing GUI와 CLI

### 6) 핵심 기능
가장 중요한 기능은 예약 생성과 중복 방지입니다.

`BookingResource`는 예약 생성 전에 다음 검증을 수행합니다.

1. 예약 번호 중복 확인
2. 고객/지점/테마 존재 여부 확인
3. 날짜와 시간 형식 검증
4. 시작 시간이 종료 시간보다 앞서는지 확인
5. 예약 시간이 5분 단위인지 확인
6. 같은 날짜와 같은 테마에서 시간이 겹치는 예약이 있는지 확인
7. 예약 인원이 테마의 최소/최대 인원 범위 안에 있는지 확인
8. 총 가격을 `인원 수 * 1인 가격`으로 계산

중복 예약은 `ReservationService.checkOverlap()`에서 `LocalTime` 기반으로 판단합니다.

```text
새 예약 시작 < 기존 예약 종료 && 새 예약 종료 > 기존 예약 시작
```

취소된 예약은 중복 검사에서 제외됩니다.

### 7) 파일 저장/불러오기
각 Service는 `ObjectOutputStream`과 `ObjectInputStream`을 사용해 Map 데이터를 파일로 저장합니다.

- `userData.txt`
- `buildingData.txt`
- `roomData.txt`
- `resData.txt`
- `playResultData.txt`

저장/불러오기 전체 흐름은 `BookingResource.saveAllData()`와 `BookingResource.loadAllData()`에서 통합 처리합니다. 파일이 없을 때는 예외로 종료하지 않고 “저장된 데이터가 없어 새로 시작합니다.”라는 메시지를 보여줍니다.

### 8) 운영 통계와 Stream API
`StatisticsService`는 Stream API를 사용해 다음 통계를 계산합니다.

- 전체 예약 수
- 예약 완료 수
- 취소 예약 수
- 완료된 플레이 수
- 성공률
- 평균 힌트 사용 횟수
- 총 예상 매출
- 테마별 예약 수
- 테마별 성공률
- 가장 인기 있는 테마
- 힌트를 가장 많이 사용한 예약

`Collectors.groupingBy`, `Collectors.counting`, `mapToInt`, `average`, `sum`, `max`를 사용했습니다.

### 9) 시연 순서
5분 발표에서는 다음 순서로 시연합니다.

1. 메인 화면 소개
2. 고객 등록
3. 관리자 이메일로 지점/테마 등록
4. 테마 상세 정보 확인
5. 예약 생성
6. 같은 시간 예약 시 중복 검증 확인
7. 플레이 결과 등록
8. 운영 통계 확인
9. 데이터 저장/불러오기 확인

### 10) 아쉬운 점과 향후 개선 방향
현재는 별도 로그인 화면 없이 관리자 기능 실행 시 관리자 이메일을 입력받는 방식입니다. 향후에는 로그인 세션, 데이터베이스, 날짜/시간 선택 UI, 더 정교한 매출 통계를 추가할 수 있습니다.

## 2. 10분 발표 버전

### 1) 프로젝트 제목
안녕하세요. 제가 발표할 프로젝트는 **방탈출 예약 관리 시스템**입니다.  
이 프로젝트는 Java 17, Swing GUI, CLI, MVC 구조, File I/O, Object Serialization, Stream API, 예외 처리, JUnit 테스트를 활용해 구현한 데스크톱 기반 방탈출 카페 운영 관리 프로그램입니다.

### 2) 개발 배경
방탈출 카페는 일반적인 회의실 예약보다 더 많은 정보를 관리해야 합니다. 단순히 시간만 예약하는 것이 아니라, 다음과 같은 운영 데이터가 함께 필요합니다.

- 고객 이메일과 이름
- 방탈출 카페 지점
- 방탈출 테마명
- 테마 장르와 난이도
- 플레이 시간
- 최소/최대 인원
- 1인 가격
- 예약 인원과 총 가격
- 예약 상태
- 플레이 성공 여부
- 힌트 사용 횟수
- 남은 시간
- 운영 통계

따라서 기존 Room Booking 프로젝트를 방탈출 도메인에 맞게 확장했습니다.

### 3) 문제 정의
처음 구조에서는 방 이름과 예약 여부 중심으로만 관리되기 때문에 실제 방탈출 예약 시스템으로 사용하기에는 부족한 점이 있었습니다.

첫째, 같은 날짜와 같은 테마에 시간이 겹치는 예약을 안정적으로 막아야 했습니다.  
둘째, 테마별 최소/최대 인원과 가격을 반영해야 했습니다.  
셋째, 예약 취소를 단순 삭제가 아니라 상태 변경으로 남겨야 했습니다.  
넷째, 플레이 이후 힌트 사용량과 성공 여부를 기록해야 했습니다.  
다섯째, 운영자가 예약 수, 취소 수, 성공률, 인기 테마, 매출을 확인할 수 있어야 했습니다.

### 4) 해결 목표
이 프로젝트의 해결 목표는 다음과 같습니다.

1. 사용자와 관리자를 구분합니다.
2. 지점과 방탈출 테마를 관리합니다.
3. 예약 생성 시 날짜, 시간, 인원, 가격을 검증합니다.
4. 같은 날짜와 같은 테마의 시간 중복 예약을 방지합니다.
5. 예약 취소와 완료 상태를 관리합니다.
6. 플레이 결과와 힌트 사용 횟수를 저장합니다.
7. 운영 통계를 Stream API로 계산합니다.
8. GUI와 CLI 모두에서 한글 안내 메시지를 제공합니다.
9. 데이터를 파일로 저장하고 다시 불러올 수 있게 합니다.
10. 서비스 계층 중심의 JUnit 테스트를 작성합니다.

### 5) 사용 기술

#### Java 17
전체 프로젝트는 Java 17 기준으로 작성했습니다.

#### Swing GUI
GUI 화면은 `src/view/GUI` 패키지에 있습니다.  
대표적으로 `Main_GUI`, `AddReservation`, `AddRoom`, `StatisticsGUI`가 있습니다.

#### CLI
CLI는 `src/view/CLI/CommandLine.java`에서 담당합니다.  
GUI 없이도 고객, 지점, 테마, 예약, 저장/불러오기 기능을 사용할 수 있습니다.

#### MVC 구조
패키지를 다음처럼 분리했습니다.

```text
model      : 데이터 클래스
service    : 비즈니스 로직, 저장/조회/검증
controller : View와 Service 연결
view       : GUI와 CLI
```

#### Collection Framework
각 Service는 `Map`과 `HashMap`을 사용합니다.

- 고객: 이메일을 key로 저장
- 지점: 지점명을 key로 저장
- 테마: 테마명을 key로 저장
- 예약: 예약 번호를 key로 저장
- 플레이 결과: 예약 번호를 key로 저장

#### Enum
두 가지 Enum을 사용했습니다.

- `UserRole`: `CUSTOMER`, `ADMIN`
- `ReservationStatus`: `RESERVED`, `CANCELLED`, `COMPLETED`, `NO_SHOW`

#### File I/O와 Object Serialization
각 Service는 `ObjectOutputStream`, `ObjectInputStream`을 사용해 Map 객체를 파일에 저장합니다.  
파일 경로는 각 Service의 `DATA_FILE` 상수로 분리했습니다.

#### Stream API
`StatisticsService`에서 Stream API를 사용합니다.  
`groupingBy`, `counting`, `mapToInt`, `average`, `sum`, `max`로 통계를 계산합니다.

#### 예외 처리
입력값이 잘못되었을 때 `IllegalArgumentException`으로 한글 메시지를 제공합니다. GUI는 `JOptionPane`, CLI는 콘솔 출력으로 사용자에게 안내합니다.

#### JUnit 테스트
`src/Tests/basic` 아래에 서비스와 모델 중심 테스트를 작성했습니다.

- GUI: `src/view/GUI`
- CLI: `src/view/CLI/CommandLine.java`

### 9) 예약 생성 흐름
예약 생성은 `AddReservation` 또는 CLI에서 시작됩니다. 이후 `BookingResource`를 통해 Service 계층으로 전달됩니다.

예약 생성 흐름은 다음과 같습니다.

1. 예약 번호 입력
2. 고객 이메일 입력
3. 지점명 입력
4. 테마명 입력
5. 예약 날짜 입력
6. 시작/종료 시간 입력
7. 예약 인원 입력
8. 예약 번호 중복 확인
9. 고객/지점/테마 존재 확인
10. 날짜/시간 형식 검증
11. 최소 예약 시간과 5분 단위 검증
12. 예약 중복 검사
13. 테마의 최소/최대 인원 검증
14. 총 가격 계산
15. 예약 저장

총 가격은 다음 공식으로 계산합니다.

```text
총 가격 = 예약 인원 × 테마 1인 가격
```

### 10) 예약 중복 방지 로직
예약 중복 방지는 `ReservationService.checkOverlap()`에서 처리합니다.

문자열 비교가 아니라 `java.time.LocalTime`으로 변환해 비교합니다.

```text
새 예약 시작 < 기존 예약 종료 && 새 예약 종료 > 기존 예약 시작
```

이 조건이 참이면 시간이 겹치는 예약입니다.  
단, `CANCELLED` 상태인 예약은 중복 검사에서 제외합니다.

### 11) 파일 저장/불러오기 방식
각 Service는 자기 데이터를 Map으로 가지고 있습니다.  
저장 시에는 Object Serialization으로 Map 전체를 파일에 저장합니다.

| 데이터 | 파일 |
| --- | --- |
| 고객 | `userData.txt` |
| 지점 | `buildingData.txt` |
| 테마 | `roomData.txt` |
| 예약 | `resData.txt` |
| 플레이 결과 | `playResultData.txt` |

저장과 불러오기는 `BookingResource.saveAllData()`와 `BookingResource.loadAllData()`로 통합했습니다.

파일이 없으면 프로그램이 종료되지 않고 다음 메시지를 표시합니다.

```text
저장된 데이터가 없어 새로 시작합니다.
```

### 12) 플레이 결과와 힌트 관리
플레이 종료 후 `PlayResult`를 등록할 수 있습니다.

저장 정보는 다음과 같습니다.

- 예약 번호
- 성공 여부
- 힌트 사용 횟수
- 남은 시간
- 직원 메모

힌트 사용 횟수와 남은 시간은 0 이상이어야 합니다.  
플레이 결과를 등록하면 해당 예약 상태는 `COMPLETED`로 변경됩니다.

### 13) 운영 통계와 Stream API 활용
`StatisticsService`는 다음 통계를 계산합니다.

- 전체 예약 수
- 예약 완료 수
- 취소 예약 수
- 완료된 플레이 수
- 전체 성공률
- 평균 힌트 사용 횟수
- 총 예상 매출
- 테마별 예약 수
- 테마별 성공률
- 가장 인기 있는 테마
- 힌트를 가장 많이 사용한 예약

Stream API 사용 예시는 다음과 같습니다.

- `filter().count()`로 상태별 예약 수 계산
- `mapToInt().average()`로 성공률과 평균 힌트 계산
- `mapToInt().sum()`으로 총 예상 매출 계산
- `Collectors.groupingBy()`로 테마별 예약 수 계산
- `max()`로 인기 테마와 힌트 최다 예약 계산
